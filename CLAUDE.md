# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Goal

A lightweight, local, cross-platform voice-to-text tool. Single binary on macOS (Apple Silicon) and Windows (amd64).
The ONLY job: hold a hotkey, speak, release, text appears at the cursor. Wherever the cursor is.

## Non-Negotiable Requirements

- **Local/offline only** — no cloud APIs, no network calls (one-time model download excepted)
- **Fast** — near-zero latency between releasing key and text appearing
- **Every error handled** — zero silent failures, no swallowed errors
- **One logging standard** — structured JSON via slog, component+operation on every entry
- **Contracts are absolute** — interfaces define exact behavior, no ambiguity
- **Cross-platform drift is a build error** — the bridge `Platform` interface is satisfied in full by every adapter; new methods break compilation on the platform that hasn't implemented them
- **Build via `make` only** — `go build ./cmd/joicetyper` directly will fail or produce a degraded binary; `make build` / `make app` is the only supported path

## Build & Test

The project is multi-package. The entry point is `./cmd/joicetyper`, NOT the repo root. The repo root is `package main` only as a four-line shim that calls `internal/launcher.Main()`.

The webview UI is generated by Vite into `ui/dist/`, which is **gitignored**, and embedded into the Go binary via `//go:embed dist`. A fresh clone has no `ui/dist/`, so:

- `go build ./cmd/joicetyper` directly fails with `pattern dist/index.html: no matching files found` (intentional fail-fast in `ui/embed.go`).
- The binary also calls `uiembed.ValidateBuiltAssets()` at startup and exits non-zero if `dist/index.html` is missing/empty/stub or `dist/assets/*.js` has no real Vite bundle. This prevents a binary from silently falling back to the native debug preferences window.
- Always build via `make build`/`make app`/`make build-windows-amd64`/`make build-windows-runtime-amd64`, which chain `frontend-build` → `npm ci` → `npm run build` → `go build`.

```bash
git clone --recurse-submodules ...        # whisper.cpp is a submodule

make setup                                # macOS: brew install portaudio cmake (Windows uses MinGW + portaudio static; see scripts/make/windows.mk)
make whisper                              # build whisper.cpp (Metal on macOS) — must run before build
make download-model                       # download whisper small model (~466MB) into the OS-appropriate app support dir
make frontend-build                       # build the Vite/React UI (runs bridge-contract-check first)
make build                                # bridge-contract-check + whisper + frontend-build, then CGO_ENABLED=1 go build ./cmd/joicetyper -> build/<goos>-<goarch>/voicetype
make app                                  # macOS .app bundle with bundled dylibs (chains build)
make test                                 # bridge-contract-check + whisper-ready, then go test -v -count=1 ./...
make verify                               # bridge-contract-check, test, frontend-build, vet-cross, verify-buildinfra
make verify-mac                           # verify + app-no-version-bump (full mac pre-PR check)
make verify-windows                       # bridge-contract-check + vet-cross + verify-buildinfra (does not require Windows host)
make bridge-contract                      # regenerate the cross-language bridge stubs (Go + TS) — run after editing Platform/Service shape
make bridge-contract-check                # fail if generated stubs are out of sync
make vet-cross                            # vet the bridge package under both GOOS=darwin and GOOS=windows
make clean                                # remove build/, whisper build artifacts, ui/node_modules
```

Run a single Go test:
```bash
go test -v -count=1 -run TestFunctionName ./...
```

The binary requires Accessibility and Input Monitoring permissions on macOS (System Settings > Privacy & Security). Permission checks use `AXIsProcessTrustedWithOptions` (Accessibility) and `CGPreflightListenEventAccess` (Input Monitoring). Windows requires the WebView2 runtime (`packaging/windows/MicrosoftEdgeWebview2Setup.exe` is bundled by the installer).

### Version policy

`VERSION` is the single source of truth. `scripts/make/version.mk` derives:

- `BASE_VERSION` = contents of `VERSION` (e.g. `1.1.112`)
- `DEV_VERSION` = `BASE_VERSION-dev+<short-sha>[.dirty]` (used when `BUILD_TYPE != release`)
- `RELEASE_VERSION` = `BASE_VERSION` (used when `BUILD_TYPE = release`)
- `DISPLAY_VERSION` is what gets baked into the binary via `-ldflags`
- `PACKAGE_VERSION` is `DISPLAY_VERSION` with `+` → `-` for filename safety

Local dev builds always carry `-dev+...`. Release targets set `BUILD_TYPE := release` and require `release-check` (clean tree + `RELEASE_TAG` matching `v$(VERSION)` and pointing at `HEAD`).

### macOS packaging & release

Accountless (no Apple Developer account required):

- `make app` — `.app` bundle with bundled PortAudio dylib, ad-hoc signed
- `make dmg` — DMG of the dev `.app`
- `make mac-dev-update-artifacts` — unsigned Sparkle-style archive + appcast under `build/macos-dryrun-update/` for shape-validation only (`EDDSA_SIGNATURE=UNSIGNED`, placeholder URLs)
- `make mac-local-release-candidate` — release-versioned `.app` with hardened-runtime ad-hoc signing using `assets/macos/JoiceTyper.entitlements`, packaged into ZIP + DMG + `SHA256SUMS` + metadata under `build/macos-local-rc/`. Validates bundle metadata (bundle ID, versions, min OS, mic usage, icon, arm64), code signature flags, dylib linkage (no Homebrew leakage), ZIP/DMG contents, checksums, and Sparkle metadata consistency. Then runs an extract+install smoke that verifies the app from both the ZIP and a temporary DMG mount.
  - `MACOS_LOCAL_RC_SMOKE_LAUNCH=1 make mac-local-release-candidate` additionally launches the installed copy and verifies the process starts.
- Accountless artifacts are **expected to be Gatekeeper-rejected** after download/quarantine. That is the ceiling without an Apple Developer account.
- CI: `.github/workflows/macos-accountless-rc.yml` runs the local RC flow on PRs without secrets.

Official Developer ID / notarized release (requires secrets, manual `workflow_dispatch` only):

- `make mac-release-preflight` / `mac-notarize-preflight` / `mac-publish-preflight RELEASE_TAG=vX.Y.Z` — fail closed when credentials are missing
- `make mac-stage-sparkle` — stage Sparkle.framework + sign_update tool (pinned `MACOS_SPARKLE_DOWNLOAD_SHA256`)
- `make mac-release-app` — Developer ID signing with hardened runtime + entitlements, plist re-rendered with Sparkle feed/key
- `make mac-notarize-release` — notarize, staple, `xcrun stapler validate`, and `spctl --assess`
- `make mac-release-archive` / `mac-appcast` / `mac-release-dmg` — Sparkle-signed ZIP, appcast, signed+notarized DMG
- `make mac-release-artifacts` — produces ZIP + DMG + appcast + `SHA256SUMS` and runs `scripts/release/macos_validate_release.sh` against them (Developer ID signature, hardened runtime, version match, Sparkle signature, appcast↔archive consistency, no Homebrew dylib leakage)
- `make mac-publish-github-release` — uploads ZIP + DMG + appcast + `SHA256SUMS` to the GitHub Release via `gh`
- CI: `.github/workflows/macos-release.yml` (manual `workflow_dispatch` only — no `push: tags:` trigger pre-credentials)

Local secrets/config live in `packaging/macos/release.env.local` (gitignored); `release.env.example` is placeholder-only.

### Windows packaging & release

- `make windows-preflight` — checks the supported Win11 toolchain (MinGW GCC/G++, cmake, mingw32-make, pkg-config/pkgconf, Vulkan SDK, Inno Setup, PortAudio source checkout)
- `make build-windows-amd64` — non-CGO Windows shell build (`-H=windowsgui`)
- `make build-windows-runtime-amd64` — full native CGO/Vulkan/whisper runtime bundle (requires `third_party/portaudio-windows-src`; auto-stages whisper DLLs and MinGW support DLLs)
- `make package-windows` — Inno Setup installer from staged runtime artifacts (no rebuild)
- `make package-windows-msix` — MSIX packaging (`packaging/windows/msix/`); test-sign and release variants exist
- Release variants (`*-release`) require `release-check` and a clean tree. CI: `.github/workflows/windows-release.yml`.

## Architecture

Multi-package Go module (`module voicetype`). The repo root is **not** the implementation — it's a shim. All non-trivial code lives under `internal/` and `ui/`.

### Core pipeline

`HotkeyListener` -> `app.App` -> `Recorder` -> `Transcriber` -> `Paster`

Component interfaces live in `internal/core/runtime/contracts.go` (package `app`). `app.App.Run` consumes a `<-chan HotkeyEvent` and drives the record→transcribe→paste pipeline with state transitions (`StateLoading`/`StateReady`/`StateRecording`/`StateTranscribing`).

### Bridge (Go core ⇄ webview UI)

The preferences/onboarding UI is a webview that talks to Go via a strict, versioned bridge:

- `internal/core/bridge/platform.go` — `Platform` interface every OS adapter must implement IN FULL (config, permissions, devices, models, hotkey capture, app state, logs, clipboard, machine info, mic monitor, input volume, …). No nullable function fields, no `if nil` branches. Genuinely platform-exclusive features still appear here; the unsupported side returns a `ContractError` (`ErrorCodeInternal`).
- `internal/core/bridge/api.go` — `Service` wraps a `Platform` and exposes the methods the router calls. `NewService(nil)` panics by design.
- `internal/core/bridge/router.go` — JSON-RPC-ish dispatcher used by the webview host on each platform.
- `internal/core/bridge/generated/` and `ui/src/bridge/generated/` — emitted by `scripts/generate_bridge_contract`. Run `make bridge-contract` after changing `Platform` or DTOs; `make bridge-contract-check` (in CI and chained from most build targets) fails on drift.
- `contracts/bridge/v1/` — versioned snapshot of the contract committed for compatibility tracking.

### Preferences UI

**Important:** the preferences window is a **webview loading the embedded React app**, not a native AppKit/Win32 form. Do not regress this.

- macOS: `internal/platform/darwin/webview.go` + `webview_darwin.{h,m}` host a `WKWebView` and serve assets out of `ui/embed.go` (`uiembed.EmbeddedAssets` from `//go:embed dist`). Bridge calls round-trip via JS message handlers into `bridge.Service`.
- Windows: `internal/platform/windows/webview.go` + `webview_host_windows.go` host the WebView2 runtime and serve the same embedded assets.
- Native settings code (`internal/platform/darwin/settings.go` + `settings_darwin.{h,m}`, `internal/platform/windows/settings.go`) still exists as a debug-only fallback. Setting `JOICETYPER_USE_NATIVE_PREFERENCES=1` (or `JOICETYPER_USE_WEB_SETTINGS=0`) flips back to the native window. The webview is the default and the surface users see (left-rail navigation: Capture / Mode / Vocabulary / Permissions / Logs / About).
- The TypeScript entry `ui/src/App.tsx` reads a bootstrap blob from `window`; if present it renders `SettingsScreen`, otherwise a placeholder shell.
- `ui/dist/` is **gitignored** and produced by `npm run build` via `make frontend-build`. Do not commit it. `ui/embed.go` enforces both compile-time (`//go:embed dist/index.html`) and runtime (`ValidateBuiltAssets()`) checks so a missing/stub UI cannot ship.

When an AI agent in another environment "rebuilds the native preferences screen": it edited `internal/platform/*/settings*.{go,m}` instead of `ui/src/settings/`. The webview path is the source of truth.

### Threading model

- Main goroutine: locked to OS thread, runs CFRunLoop on macOS / message pump on Windows (hotkey listener blocks here)
- App.Run goroutine: processes `HotkeyEvent` channel
- recorder readLoop goroutine: one per recording session, reads PortAudio buffers
- Transcription goroutine: async transcribe+paste after key release
- Webview host: platform-specific event loop integration; bridge calls dispatched via `bridge.Router`

### cgo / Objective-C / Win32

Platform adapters use cgo. macOS code lives in `internal/platform/darwin/*_darwin.{m,h}` linking CoreGraphics, Carbon, Cocoa, AppKit, WebKit, Metal, Accelerate. Windows code lives in `internal/platform/windows/` linking Win32, Shell, WebView2, WASAPI/`go-wca`, `go-ole`. The whisper.cpp library is statically linked from `third_party/whisper.cpp/build/`.

## Technical Context

- **Platforms**: macOS Apple Silicon (arm64) and Windows amd64
- **Language**: Go 1.26+ with cgo; TypeScript/React 18 + Vite for the UI
- **Speech model**: whisper.cpp `small` (~466MB), Metal GPU on macOS, Vulkan/CPU on Windows
- **Audio capture**: PortAudio via `gordonklaus/portaudio`; WASAPI via `go-wca` for Windows device enumeration / volume
- **Webview**: `WKWebView` (macOS), `wailsapp/go-webview2` + WebView2 runtime (Windows)
- **Updater**: Sparkle 2.x on macOS (release-only, requires `MACOS_SPARKLE_PRIVATE_KEY_FILE`)
- **Submodule**: `third_party/whisper.cpp` (clone with `--recurse-submodules`)
- **Config**:
  - macOS: `~/Library/Application Support/JoiceTyper/config.yaml`
  - Windows: `%APPDATA%/JoiceTyper/config.yaml`
- **Default hotkey**: Fn+Shift on macOS (configurable per platform via the Capture pane)
- **Decode modes**: `beam` (better sentence quality) or `greedy` (lower latency)
- **Punctuation modes**: `off`, `conservative`, or `opinionated`
- **Vocabulary**: User-defined words/phrases in config, passed as whisper `initial_prompt` to bias recognition
- **Translation**: optional, wired end-to-end through the bridge (toggle visible in the UI badge "Translation active")

## Engineering Standards

- Every Go error checked and handled with `component.operation` context format
- Structured logging: `log/slog` JSON, always `component` + `operation` fields
- No `fmt.Println` (except CLI helpers like `audio.ListInputDevices`), no `log.Printf`, no silent error swallowing
- Interfaces at every component boundary: `app.HotkeyListener`/`Recorder`/`Transcriber`/`Paster` in `internal/core/runtime/contracts.go`, `bridge.Platform` in `internal/core/bridge/platform.go`
- Tests use mocks at interface boundaries; bridge platform tests use `funcPlatform{}` (NOT nil)
- Error format: `"component.operation: description: %w"` (e.g., `"recorder.Start: open stream: %w"`)
- Cross-platform changes: edit the `Platform` interface, regenerate the contract (`make bridge-contract`), implement on every adapter, and run `make verify` (or at minimum `make vet-cross`) before pushing
- `internal/buildinfra/` Go tests are the executable spec for Make targets, release flow ordering, workflow shape, etc. — touching the build/release pipeline means updating these tests, not bypassing them

## Working in this repo — DO NOT

- Do **not** treat the repo root as the implementation. The entry point is `./cmd/joicetyper`; `internal/launcher.Main()` does the real work.
- Do **not** run `go build ./cmd/joicetyper` directly when the frontend hasn't been built. Use `make build`/`make app`/`make build-windows-runtime-amd64`. The compile-time `//go:embed dist/index.html` and runtime `ValidateBuiltAssets()` will refuse to start a binary without the real Vite bundle — that is intentional, do not weaken those checks to "make it compile".
- Do **not** commit `ui/dist/`. It is gitignored, churny, and reproducible from `ui/src/` via `make frontend-build`. Same for `ui/node_modules/`, `build/`, and `third_party/whisper.cpp/build/`.
- Do **not** add a new native-AppKit/Win32 preferences pane to "fix" the settings UI. The settings UI is the embedded React app in `ui/src/settings/`. If a pane needs a new field, add it to a `Pane.tsx`, extend `bridge.Platform` + DTOs, regenerate the contract, and implement on each platform.
- Do **not** add a new bridge call by editing only one platform adapter. The `Platform` interface is satisfied in full or the build breaks — that's the contract.
- Do **not** edit the generated files under `internal/core/bridge/generated/` or `ui/src/bridge/generated/` by hand. Run `make bridge-contract`.
- Do **not** add network calls outside the model-download path (allowlisted in `transcription`).
- Do **not** weaken release-pipeline assertions in `internal/buildinfra/` to "get tests green". Those tests guard against the failure modes that have actually shipped before (missing checksums, dev-versioned release artifacts, auto-tag-triggered notarization with no secrets, etc.).
- Do **not** auto-trigger the official macOS release workflow on tag push without Apple credentials configured. It is `workflow_dispatch`-only on purpose.
